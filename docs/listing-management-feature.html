Listing Management — Profile Closet Owner Actions

Technical reference for the closet → detail → edit / mark sold / delete feature  ·  SwapHaven API  ·  Jul 2026

Scope: A seller opens their own listing from the profile closet and can edit listing fields, mark as sold (traded / cash / given away), or delete the listing. Status changes never go through PATCH — they use POST /sold or DELETE. All mutating routes require auth and owner check (listing.userId === token.sub).

1 · Feature overview

The profile closet shows a grid of the user's non-deleted listings (status != 'deleted'). Tapping a card opens the listing detail screen. When the viewer is the owner, the app shows owner actions: Edit, Mark Sold, Delete.

ScreenPrimary APINotes
Profile closet grid GET /api/users/:userId/listings Public. Returns items, total, cursor pagination.
Listing detail (owner) GET /api/listings/:listingId Public. Includes seller snapshot. Client compares listing.userId to session.
View count ping POST /api/listings/:listingId/view Auth. Fire-and-forget 204; increments view_count.
Edit listing PATCH /api/listings/:listingId Auth + owner. Title, value, condition, category, description, trade wants.
Photo add / remove POST / DELETE …/images Auth + owner. Presigned upload via POST /api/media/presign first.
Mark as sold POST /api/listings/:listingId/sold Auth + owner. Sets status = traded, records sold_method.
Delete listing DELETE /api/listings/:listingId Auth + owner. Soft-delete (status = deleted). 409 if already deleted.
Product note — Mark Sold UI: The mobile app sends only { soldMethod, shareWin }. The API accepts optional tradedWithUserId but the "Who did you trade with?" field is intentionally omitted from the UI.

3 · Listing status lifecycle

Listings use enum listing_status: active, traded, paused, deleted. Owner actions in this feature transition active → traded (mark sold) or active → deleted (delete).

sold_method values (set only on POST /sold):

  • traded_on_barter — increments seller user_profiles.total_trades
  • sold_for_cash
  • given_away

5 · Database schema

Core tables touched by listing management. Migration 0012_listing_sold_meta.sql adds sold columns.

Deployment: Code referencing sold_method requires migration 0012_listing_sold_meta.sql on the target database. Missing columns cause 500s on any query selecting from listings (e.g. swipe deck, closet).

POST /api/listings/:listingId/images Auth { url, position? } 201 image row
DELETE /api/listings/:listingId/images/:imageId Auth — 204

PATCH — fields intentionally excluded

These exist on the listing row but are not editable via PATCH (Edit Listing screen):

  • status — use POST /sold or DELETE
  • locationCity, location* — set at create time only
  • acceptCashTopUps, isSwipeOnly — not on edit screen
